## 结果

10 条显式包含 `is:public` 的 GitHub 查询得到 188 个去重来源。平衡抽样先为
每条查询保留至少两个结果，再按全局信号补齐。20 个抽样来源全部取得不可变
revision，并成功完成递归目录树预检：14 个多 Skill 套件、1 个单 Skill 来源、
5 个未检测到 `SKILL.md` 的来源，另有 6 个来源存在需要独立评审的 Agent Hook
表面。

这修复了旧通用发现路径的两个问题：即使使用已登录 GitHub，查询也不能混入
私有仓库元数据；低星但与短板高度相关的来源，也不会再被全局热门却无关的仓库
挤出结构预检。

## 预检处置

预检明确拒绝“整仓即一个候选”的思路。ECC、Ruflo、
agentic-awesome-skills 等大型集合，后续必须按组件核对来源、原子性、重叠、
冗余、安全和更新影响；发现 Skill 也不等于批准套件中的 Hook。CC Switch 被
保留为运营管理器基线，不误分类为 Skill 来源；没有 `SKILL.md` 的普通查询噪声
退出当前 Skill 车道。

只有以下 5 个来源进入“不执行的静态复核”：

- `Forward-Future/loopy`：较小的自适应循环套件；
- `cbcraftlab/governed-skill-tree`：面向协作短板的治理套件；
- `sanjay3290/ai-skills`：跨 Agent 套件；
- `diet103/claude-code-infrastructure-showcase`：宿主特定的自动激活与 Hook
  套件；
- `mksglu/context-mode`：包含上下文、记忆、路由、MCP 与 Hook 的系统，
  许可证分类尚未解决，可执行面也较大。

进入下一步只表示元数据和结构值得继续读取固定 revision 的公共源码；不代表
质量、优越性、短板覆盖、采用、下载、执行、安装或 Hook 决策。

## 短板与标准化边界

候选与 STM/P 坐标的联系，只是依据描述和结构形成的假设。它不能证明某来源
真的弥补短板、优于原生路径或产生正净收益。静态复核仍要把候选与原生、官方、
运行时、已安装、组合方案、无 Skill 和无 Hook 路径比较。

当前查询字段、筛选规则、Harness 档位和预检项只是可修正的工作假设与观测
协议，不是硬标准。只有当价值在多个独立来源、Agent 或宿主、任务类别以及真实
反馈周期中稳定复现，并且收益大于上下文、控制和维护成本，再经过单独的权威与
采纳决策，才有资格抽象为硬标准。
