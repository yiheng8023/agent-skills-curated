# 第三轮权限边界需求复核 — 2026-07-18

这条有来源支持的需求线绑定 `STM-20`、`P9` 与 `SG-05`：当 Agent 越过已绑定的
读取、写入、安装、Git、账号、发布、删除、部署、记忆、状态、数据、成本或影响范围
边界时，过度代理就转化为权限边界失败。固定在 `e060a08` 的 CALIBRATION 语料只是
只读候选证据，不是 Skill 规格、运行时权限模型或通用标准。

当前分层路径足以覆盖这条有边界需求：原生推理只执行明确授权的低风险动作；
`intent-contract` 在路由前绑定权限；`capability-router` 在高风险、权限扩张、实质成本、
冲突或歧义处停到人类确认；已批准的 `git-guardrails` 只适用于用户明确请求的窄 Git
拦截面；`closure-contract` 保护收尾和状态声明；回滚 recipe 提供恢复支持；真正的
沙箱、凭据、审批、事务、撤销和审计强制仍归宿主运行时；后果性判断仍归有正当决策权
的人类或领域权威。10 个确定性场景选择 `human-authority`，其中 9 个直接覆盖权限边界，
整体路由语料继续保持 105/105 通过。

三层不能混为一谈：Skill 或 Hook 可以提醒、收窄、拦截已知表面或发起询问，但不能
凭空创造运行时权限，也不能替代负责主体。当前证据因此没有证明残余 Skill 缺口，也不
要求外部发现或自研；不授权候选执行、准入、运行时控制、通用审批硬标准、真实环境
改动、提交或推送。只有出现已记录的控制失败、有边界的宿主强制缺口，或实质不同的
候选/来源证据时才重新评估。
