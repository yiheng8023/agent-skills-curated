# CC Switch 可丢弃来源更新与恢复复核——2026-07-18

## 结论

本次用本机回环 HTTP 服务向 CC Switch 3.17.0 的临时源码副本提供兼容 GitHub 路径的
v1、v2 仓库归档。夹具验证了来源型安装、按哈希发现更新、更新前备份、替换为 v2，
以及手工恢复 v1。显式让一次性配置目录取得最高优先级后，目标 Rust 测试通过。

故障注入确认了真实事务缺口：`update_skill` 会忽略备份创建错误，先删除旧 SSOT，
再复制新来源，而且没有自动回滚。如果删除后复制失败，系统可能留下“SSOT 已不存在，
旧数据库行仍存在”的不一致状态。恢复路径已经验证：先通过现有卸载路径清理精确匹配的
陈旧记录，再从仍可读取的更新前备份恢复 v1。

## 隔离事故

隔离修正前运行的诊断测试曾短暂在真实 CC Switch Home 中创建 2 条纯测试 Skill 行、
1 条纯测试来源行、1 个 `demo` SSOT 目录和 3 个 fixture 备份目录。原因是 Windows
上的另一个隔离缺陷：
当一次性数据库尚不存在时，`get_app_config_dir` 会让旧版 `HOME` 数据库回退覆盖显式的
`CC_SWITCH_TEST_HOME`。

清理过程采用失败关闭。第一次事务只有在 fixture 身份、来源仓元数据、描述、内容哈希、
精确目录树和备份元数据全部匹配时才删除。后续全行审计发现两条更早的纯数据库测试
记录；第二次事务要求两条记录逐字段匹配，且一旦发现同名载荷目录就中止。一致性数据库
副本均保留在 `C:/tmp`，SHA-256 分别为
`F0A041FFB6AF2103D5874AFC899CC8808FA863876704A520AE7108232B203C6D` 和
`EF2EE5728842E3BFF594EE328C91AB511C0187F4CDC51475A0286D61C758F209`。

最终清理后，纯测试 Skill 行为 0、纯测试来源行为 0、fixture 备份为 0，不存在 `demo`
或 `duplicate-skill` SSOT，也没有 Agent 投影；真实数据库回到 248 条 Skill、5 条来源，
CC Switch Skill 存储恢复为 514 个文件。修正测试 Home 优先级后的第二次运行没有新增
真实 fixture 残留。

## 修订后的金丝雀门

CC Switch 仍是选定的运行管理器，但它的更新器不是原子事务。第一次真实金丝雀必须只选
一个已审查 Skill，先取得一致性数据库备份、精确前态哈希和可独立读取的 Skill 备份，
预览所有权转换，并演练明确的失败恢复。第一次金丝雀不允许自动更新，也不能把当前
`update_skill` 路径当作回滚证据。

本次没有迁移、替换、更新、禁用或恢复任何真实用户 Skill，也没有修改 Hook、MCP、
Plugin、App、指令载体、消费者配置、上游仓库、提交或远端。
