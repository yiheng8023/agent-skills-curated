# 动态运行时控制缺口复核 — 2026-07-18

## 当前判断

MCP 负载问题值得继续研究，但还不能直接推出“必须自研 Skill 或 Hook”。本任务中的 Codex 0.144.5 可见 13 个 MCP 条目，全部显示启用。CLI 支持 list、get、add、remove、login、logout，却没有专门的启停命令；抽样的插件/运行时可见名称也无法被 `mcp get` 管理。

OpenAI 官方源码在快照 `2895d82b5e449407712439ba4f89954f3fa0c7e3` 中已经提供一些启动期控制：普通 MCP 和插件 MCP 都有 `enabled` 字段，`enabled=false` 会跳过初始化，工具还可用白名单/黑名单缩减；插件本身也可禁用。但当前 `ConfigProfile` 并不包含 MCP 或插件映射，本次也没有找到会话中安全热切换、卸载已运行 MCP 的证据。

因此最小充分路径是先用原生启动期过滤。未来 Skill 可以负责“任务需要哪些能力”的判断，Hook 可以负责事件观察或启动准备；在 Codex 暴露受支持的执行与验证接口前，两者都不应宣称能可靠控制进程生命周期。

## 指令载体双版本

产品中立版本留在本仓，定义协作不变量、动态检查合同、Hook 的 `off`/`auto`/`on`、验证、回滚和安全降级。

私有 `codex-user-config` 版本承载你的 Codex 路径、主动分支/工作树与上下文偏好、真实 Hook 接线和回滚。公共用户配置默认不再手工维护第三份全文；未来确有公共价值时，只发布带来源说明、明确非权威的脱敏生成模板。
