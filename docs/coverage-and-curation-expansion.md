# Coverage And Curation Expansion

This repository should not stay limited to software engineering. Its role is to
curate reusable human-agent collaboration Skills across digital work contexts:
daily productivity, office work, software engineering, business, education,
academic and research work, creative production, analytics, operations, and
security/privacy/compliance.

The expansion path is deliberately iterative and reuse before build:

1. Bind dated demand, shortfall, risk, lifecycle, or coverage evidence.
2. Measure native, official, runtime, curated, installed, and authorized
   baselines under explicit host conditions.
3. Consume broad radar signals only inside a bounded discovery scope, then
   cluster, deduplicate, shortlist, and record why the round can stop.
4. Fix exact source pins and license/provenance boundaries, then deep-review
   representatives and compare native, single-Skill, composed
   Skill, non-Skill Harness, project-standard, and human-authority paths.
5. Prove or reject the residual gap before adaptation or repository authoring.
6. Govern, validate, admit, release, refresh, deprecate, replace, or retire the
   smallest justified capability set.
7. Accept optional consumer evidence and extract repeated standard candidates
   for separately authorized `YIYUAN-CALIBRATION` delivery.

The numbered list projects the core dependencies; it is not a universal linear
execution order. Consumer work is an optional branch, lifecycle metabolism is
cross-cutting, and standards extraction is a repeated-evidence conditional
branch. Safe parallel work is bounded inside a stage and joins before the next
decision gate.

Local runtime alignment is optional and consumer-owned. Round 02 contains a
dated authorized sync record, but that historical transaction is not a
mandatory stable program stage and does not prove current live parity.

The machine-readable coverage model lives in
`registry/collaboration-domain-coverage.json`. It separates general Skills from
specialist Skills so the repository can grow without forcing every capability
into one shape.

The iterative control plan lives in `registry/curation-expansion-rounds.json`.
The overall execution plan lives in `registry/curation-program-plan.json` and
is explained in `docs/curation-program-plan.md`.
The current machine program separates stable operating lanes, sequence gates,
bounded initiatives, historical rounds, optional consumer transactions, and
external calibration handoffs. Program completeness reconciliation must close
before Round 02 closeout is finalized, and Round 02 closeout must close before
the planned capability-survey initiative starts external discovery execution.
The continuous curation harness and upstream input boundary are explained in
`docs/curation-harness-model.md`.
The round lifecycle contract lives in
`registry/round-lifecycle-contract.json` and is explained in
`docs/round-lifecycle-contract.md`. Candidate radar and coverage records are
not approval. A candidate becomes runtime-relevant only after source pinning,
license/provenance review, security review, portability review, overlap
resolution, adaptation, acceptance, repository verification, and stage
closeout.
