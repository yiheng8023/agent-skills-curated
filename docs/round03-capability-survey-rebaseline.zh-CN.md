## 建议工序

```text
绑定有证据的 STM / P / SG 需求坐标与边界
-> 测量带日期的原生 / 官方 / runtime / 已安装能力基线
-> 进行有边界、只读的公开元数据发现
-> 聚类、去重并选择最少代表样本
-> 固定来源后深度审查，但不执行候选代码
-> 比较原生 / 单 Skill / 组合 / 非 Skill Harness 路径
-> 判断已覆盖、残余缺口成立或缺口尚未证明
-> 仅当残余缺口成立时，再比较仅 Skill / Skill + Hook /
   Hook 或其他非 Skill 控制 / 项目硬标准 / 人工检查点
-> 形成十项结果包与下一轮候选
```

当新增候选基本只落入既有聚类，且不再实质改变覆盖判断、架构备选、候选处置或残余
缺口证据时，本轮发现应停止，并记录剩余不确定性和重新检查触发条件。

Hook 只在重要残余缺口已经成立后才进入备选，不是每个自制 Skill 的默认附件。只有当
事件触发召回或有边界的前置/后置检查确有必要、宿主 Hook 面可检查且可回滚，并且收益
高于噪声、延迟、权限、上下文和维护成本时才采用。Hook 仍属于消费端的召回或约束辅助，
不是权威源，也不是证据源。

## 边界

- 本重基线被接受后，可以读取公开 GitHub 元数据和完成分类所需的最少来源文本；
- 不执行候选代码、脚本、依赖、Hooks、命令或具备网络行为的载荷；
- 不连接账号或 OAuth，不暴露凭据，不付费，不扩展私有数据边界，不修改 live runtime，
  不跨仓写入，不执行 Skill 准入、发布清单变更、标准晋升或远端推送；
- Codex 单一宿主或弱模型测试证据不能被包装成通用跨 Agent 证明。
