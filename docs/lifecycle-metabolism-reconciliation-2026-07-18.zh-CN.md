# 生命周期代谢对账——2026-07-18

本对账把“已经工作的反馈回流路径”“确定性的已批准 Skill 退休合同”和“仍待
补充的真实消费者证据”明确分开。

## 已验证的部分

仓库已有一次真实消费者反馈循环。`codex-user-config` 验证选定批次后，1 个候选
保留为 Recipe 投影，2 个候选合并进既有已批准 Skill，没有新增独立 Skill。
结果随后回流到发现层，用于抑制同一批精确候选的重复提议，同时不把整个上游仓库
视为已批准或已拒绝，并选择“先暂停观察，再决定下一批”。

生命周期政策和 Harness 现在绑定六类事件：消费者、社区、安全、许可证或来源、
上游刷新、验证。任何一类都可以重新打开复核，但不会自动替换最后一个已知良好
版本。PM Skills 还提供了一次已观察到的只读上游版本增量复核。

这些证据足以验证本仓治理的反馈回流路径，但不宣称六类触发器都已经在生产中发生。

## 确定性退休合同

现在有 8 个政策夹具：5 个正向结果覆盖保留、替代、弃用、回滚和退休；3 个
反向结果拒绝自动上游替换、缺少迁移与回滚的退休、以及用反馈直接批准候选。
这验证了本仓合同能够支持更新、替代、弃用、迁移、回滚和退休，而且无需修改
真实运行时。

自研 Manager 项目继续只作为非载荷的历史证据。当前仍没有真实的已批准 Skill
消费者完整走过退休路径，因此“已观察到的运行时成熟度”和“跨消费者有效性”仍是
开放主张；但本仓政策与确定性夹具范围的验收已经验证。

## 权限边界

反馈记录可以重新打开复核或抑制精确重复提议，但不能批准候选、修改发布、覆盖
消费者、安装 Skill，也不能改变 Hook 或运行时。发布和消费者变更仍保留独立的
权限与验证闸门。
